# Fix initial Agent Host session list display in Agents app sidebar

**Date:** 2026-04-20
**VS Code branch:** roblou/agents/fix-agent-host-session-display
**VS Code SHA at finalize:** d05eca7455
**PR:** https://github.com/microsoft/vscode/pull/311293

## What was done

Agent Host sessions returned by `listSessions()` were not visible in the Agents-app sidebar (`SessionsList` in `src/vs/sessions`) on a fresh launch. They only appeared after the user sent a message to the agent host — at which point a `notify/sessionAdded` would force the renderer to re-fetch, and the entire list would suddenly populate. Reload happened to mask the bug because the auth token was already cached.

Fixed by changing `CopilotAgent.listSessions()` and `_listModels()` to stop silently returning `[]` when `_githubToken` is unset. Both now go through `_ensureClient()`, which throws `ProtocolError(AHP_AUTH_REQUIRED, ...)` (-32007) — the AHP-spec-mandated response for any command issued against an agent declaring `protectedResources` with `required: true`. The renderer side already had `LocalAgentHostSessionsProvider`'s constructor `autorun(authenticationPending)` (added in [the loading-indicator session](../2026-04-17-session-loading-while-authenticating/summary.md)) which fires `_refreshSessions()` once auth settles; that's the natural retry trigger for the silent-listSessions case. The blanket `catch` in `_refreshSessions` swallows the throw, which is intentional — we do NOT want to pop a sign-in dialog just to render the sidebar.

Two new tests in `localAgentHostSessionsProvider.test.ts`: `eagerly populates and fires onDidChangeSessions after construction without a getSessions() call`, and `defers eager session list fetch until authentication settles`. The pre-existing `returns empty models and sessions before authentication` test was renamed and updated — it had been pinning the wrong behavior as a feature.

## Key decisions

- **Throw `AHP_AUTH_REQUIRED` rather than block forever.** The user articulated the values rule clearly: "I don't think a method should just block forever. It's more clear to throw the error and then the client knows to retry and it knows why it failed." A protocol response should never be ambiguous between "I don't know yet" and "I know, and the answer is empty." The throw makes the precondition explicit and gives the client a typed signal to retry off.
- **Design for the protocol contract and the in-tree client, not for theoretical external clients.** When weighing fixes, the user pushed back: "I don't care about what other callers think right now, at least other callers that are not in this codebase. I care about what makes sense for the contracts of the protocol and the server and so on." This is a load-bearing principle: AHP is an open protocol that other implementations could in theory consume, but designing for hypothetical consumers leads to over-flexible, under-decided APIs. The bar is "what does the spec say, and what does our concrete client need." If a future external client appears, *they* adapt to the contract, not the other way around. This eliminated several alternative shapes (e.g. "make `listSessions` configurable to return-empty-or-throw") that would have been tempting if we were optimizing for unspecified third-party callers.
- **Fix at the agent layer, not the consumer (cache) layer or the renderer auth-pending layer alone.** Three candidate fixes were considered: (1) make `SessionsManagementService` call `provider.getSessions()` after subscribing, (2) make the local provider's eager-load wait on `authenticationPending`, (3) make the agent throw `AHP_AUTH_REQUIRED` instead of returning `[]`. (1) was wrong because it pushes provider-specific lazy-cache knowledge into a generic consumer. (2) worked in isolation but masked the root cause — the protocol contract was being violated. (3) is the contract-correct fix; (2) becomes the natural retry trigger that pairs with it.
- **Don't pop a sign-in dialog just to list sessions.** `agentHostSessionHandler.ts` already has an `_isAuthRequiredError` pattern that, on `createSession` failure, prompts the user via `resolveAuthentication`. We deliberately do NOT do that for `listSessions` — interactively prompting just to render the sidebar would be terrible UX. The autorun-on-`authenticationPending` retry strategy is the right one for read-only background queries.
- **Apply the same fix to `_listModels()`.** Same `!_githubToken` early-return; same protocol violation. `_refreshModels()` was the only direct caller and already had a guard + try/catch, so the change is invisible to it but corrects the contract for any future consumer.

## What went wrong or was misunderstood

- **Initial fix attempt was at the wrong layer (consumer-side `SessionsManagementService.getSessions()` after subscribe).** Assumed the lazy-cache pattern itself was buggy. The user immediately corrected: "should be fixed on the agent host side." — **prevented by:** new doc-body section "One-shot `_ensureSessionCache` + auth-aware eager load" in `agent-host-sessions-providers.md`, which now explains the cache shape and why the fix layer is the agent (or, when the agent is correct, the auth-aware autorun on the renderer side) rather than a generic consumer.
- **Second fix attempt (eager `_ensureSessionCache()` in the local-provider constructor) "worked" on reload but not fresh launch.** Assumed eager loading was sufficient without considering the auth race. The user: "ran with your fix and it still doesn't work … When I just reload the window then it works. That makes me think it's an auth race." — **prevented by:** new gotcha on `agent-host-sessions-providers.md` calling out that the cache is one-shot AND the underlying agent must answer truthfully — the two together drive the requirement for an auth-aware trigger.
- **`CopilotAgent.listSessions()` silently returned `[]` when no token — and there was a passing test pinning that as the "expected" behavior.** The test name was `returns empty models and sessions before authentication`. A future agent reading that test could reasonably assume the silent-empty was intentional. — **prevented by:** new gotcha on `copilot-agent-provider.md` explicitly forbidding the `return []` shortcut, plus the renamed test now pinning the throw, plus the AHP-level rule on `agent-host-protocol.md`.
- **Didn't recognize the AHP spec violation upfront.** The protocol-spec reading came late; until then I was framing this as "should we wait or throw?" when the spec already mandates the answer. — **prevented by:** new "Authentication errors are explicit, not empty responses" entry in the `agent-host-protocol.md` "Patterns and gotchas" section, which makes the rule discoverable without reading the external `agent-host-protocol` repo's `docs/specification/authentication.md`.
- **Tooling miss: the renderer-side test for `defers eager session list fetch until authentication settles` was nice-to-have; the real value test (auth-required throw recovery) lives at the agent layer.** Spent time building both. The agent-layer test would have been enough on its own once the contract was fixed. Not a major loss, but a reminder that "where the bug was caused" and "where the test should live" are the same place.
- **Initially proposed multiple fix shapes biased toward "what would arbitrary callers want?"** When the user asked for a cleaner alternative, I offered three options that included shapes optimizing flexibility for hypothetical external clients. The user corrected: "I don't care about what other callers think right now, at least other callers that are not in this codebase. I care about what makes sense for the contracts of the protocol and the server and so on." — **prevented by:** new "Designing for the spec and the in-tree client, not for theoretical external ones" section in `agent-host-topology.md`. This is the load-bearing principle and now lives at the topology level so future agents read it before any contract decision.

## What we learned

- **The AHP authentication contract is in the external spec, not the local docs.** `docs/specification/authentication.md` in the sibling `agent-host-protocol` repo prescribes `AuthRequired` (-32007) as the response for `required: true` resources. Worth reading once at the start of any auth-adjacent investigation rather than inferring from the in-tree code (which had a violation).
- **A response shape that conflates "I don't know" with "I know, and the answer is empty" will eventually break a downstream cache.** The Agents-app sidebar's one-shot `_ensureSessionCache` was the cache that broke first. There may be others (model picker? customization picker?) that will surface the same way as code grows. Throwing makes the precondition explicit and unmissable.
- **Sticky observables (`authenticationPending`) compose well with one-shot caches.** The autorun fires exactly once per fresh launch (at the moment auth settles, or immediately if auth was already settled). No need for retry counters, debouncing, or polling. Pattern worth reaching for when "do this work once, but only once the prerequisite signal has arrived" is the requirement.
- **An existing test pinning the wrong behavior is one of the more dangerous failure modes.** A red test would have signaled "don't do this." A green test labeled `returns empty … before authentication` actively recruits future agents into preserving the bug. When fixing a contract bug, hunt for tests that codify the bad behavior and update them with the contract change.
