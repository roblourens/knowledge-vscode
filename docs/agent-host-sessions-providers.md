# Agent Host Sessions Providers

_Covers: src/vs/platform/agentHost/common/tunnelAgentHost.ts, src/vs/sessions/contrib/providers/agentHost/browser/baseAgentHostSessionsProvider.ts, src/vs/sessions/contrib/providers/agentHost/browser/localAgentHostSessionsProvider.ts, src/vs/sessions/contrib/providers/agentHost/browser/localAgentHost.contribution.ts, src/vs/sessions/contrib/providers/agentHost/browser/agentHostSettings.contribution.ts, src/vs/sessions/contrib/providers/agentHost/browser/agentSessionSettings.contribution.ts, src/vs/sessions/contrib/providers/remoteAgentHost/browser/remoteAgentHostSessionsProvider.ts, src/vs/sessions/contrib/providers/remoteAgentHost/browser/remoteAgentHost.contribution.ts, src/vs/sessions/contrib/providers/remoteAgentHost/browser/tunnelAgentHost.contribution.ts, src/vs/sessions/contrib/providers/remoteAgentHost/browser/webTunnelAgentHostService.ts, src/vs/sessions/contrib/providers/remoteAgentHost/electron-browser/tunnelAgentHostServiceImpl.ts, src/vs/sessions/common/agentHostSessionsProvider.ts, src/vs/sessions/services/agentHost/common/agentHostEnablementService.ts, src/vs/sessions/services/agentHost/browser/agentHostEnablementService.ts, extensions/copilot/src/extension/chatSessions/copilotcli/node/copilotcliSessionService.ts_

`LocalAgentHostSessionsProvider` and `RemoteAgentHostSessionsProvider` are the agent window's (`/sessions/`) view of Agent Host sessions. Both extend a shared abstract base, `BaseAgentHostSessionsProvider`, that owns ~all of the structural behaviour: the session cache, the three config caches, the lazy `SessionState.config` subscription seeding, AHP notification/action handlers, `sendAndCreateChat`, and a single concrete `AgentHostSessionAdapter` (`ISession` implementation). The subclasses contribute only the bits that genuinely differ: which connection to use, how to label sessions, how to map session types ↔ resource schemes, how to pick a working folder, and (remote only) connection lifecycle. Both implement `IAgentHostSessionsProvider` (defined in `src/vs/sessions/common/agentHostSessionsProvider.ts`), which extends `ISessionsProvider` with the Agent Host extras the agent window needs: dynamic session config, optional remote connection status, and an output channel id. The local provider talks to `IAgentHostService` (utility-process MessagePort); the remote provider talks to an `IRemoteAgentHostConnection` over WebSocket / SSH / tunnel relay.

For where this layer sits in the broader topology, see [agent-host-topology](./agent-host-topology.md). For the AHP wire model these providers consume, see [agent-host-protocol](./agent-host-protocol.md). Turn execution and chat rendering happen elsewhere — see [agent-host-session-handler](./agent-host-session-handler.md).

## Responsibilities

For each provider:

- **List sessions.** Calls `connection.listSessions()`, maps to `ISession` for the agent window, and maintains an in-memory `_sessionCache` keyed by raw session id.
- **Open / activate sessions.** Translates a Sessions-app session id (chat-session-style id) ↔ canonical AHP `AgentSession.uri(provider, rawId)`.
- **Create draft sessions with final resources.** New Agent Host drafts use final-looking chat resources immediately (`/${uuid}` under the host-specific resource scheme). Draft-ness is tracked by provider state (`SessionStatus.Untitled`), not by an `/untitled-*` path. The provider now eagerly creates a provisional backend session when the folder is picked, holds a session-state subscription while the new-chat form is open, and either graduates or disposes that backend session when the user sends or abandons it.
- **Surface dynamic session config.** Resolves the per-agent config schema for new sessions and exposes per-session config values to the picker UI.
- **Persist session config across windows.** Cooperates with `AgentService` and `AgentSideEffects` so a session opened from the cached list — possibly in a fresh window — sees the same picker values it had on the server.
- **Bridge connection status.** Remote provider exposes `connectionStatus` and `remoteAddress`; local provider has neither.
- **Expose changesets.** `AgentHostSessionAdapter` creates the shared changeset model consumed by the Changes view, so branch/uncommitted/all/last-turn review choices can be contributed without an Agent Host-only review surface.
- **Represent virtual remote workspaces honestly.** Remote providers carry workspace capability and tunnel connect/disconnect hooks through `IAgentHostSessionsProvider`, letting host filtering and workspace selection reflect online/offline transport state without pretending the workspace is local.

The providers do **not** own turn dispatch, file edits, terminals, permissions, or model selection — those are the [Session Handler](./agent-host-session-handler.md)'s job. The split is clean: **providers serve list and picker; handler runs turns**.

## Session enablement: `IAgentHostEnablementService`

Whether Agent Host is enabled at all is now read through `IAgentHostEnablementService` (`src/vs/sessions/services/agentHost/common/agentHostEnablementService.ts` + `browser/agentHostEnablementService.ts`) rather than by reading `AgentHostEnabledSettingId` off `IConfigurationService` at each call site. The service exposes a single `readonly enabled: boolean`, captured **once** at construction time (`configurationService.getValue<boolean>(AgentHostEnabledSettingId) ?? false`) — it is **not** reactive. `LocalAgentHostContribution`'s constructor now early-returns based on `_agentHostEnablementService.enabled` instead of `configurationService.getValue(...)`, and `LocalAgentHostSessionsProvider.supportsQuickChats` reads the same flag. Because the value is captured once, toggling `chat.agentHost.enabled` requires a reload to take effect — this removed `LocalAgentHostSessionsProvider`'s `onDidChangeCapabilities` emitter (which used to fire on a live config-change event for this setting), since there is no longer a live value to change. If a future feature needs to react to enablement changing without a reload, it needs a different, genuinely-reactive signal — don't assume `IAgentHostEnablementService.enabled` will ever update after construction.

## Coexistence with the extension-host provider

When `chat.agentHost.enabled` is `true`, `LocalAgentHostSessionsProvider` coexists with the extension-host `CopilotChatSessionsProvider` — both register with `ISessionsProvidersService` simultaneously. This is safe because each provider filters to its own sessions:

- **Local agent host** — `CopilotAgent.listSessions()` uses a database-existence ownership gate: only SDK sessions with a per-session Agent Host SQLite DB are included (see [copilot-agent-provider § Session Ownership](./copilot-agent-provider.md#session-ownership)).
- **Extension-host CLI** — new/resumed Agent Host sessions carry SDK `clientName: 'vscode-agent-host'` and are rejected before adaptation. Legacy ambiguous sessions fall back to one bulk read of `<userDataPath>/agentSessionData`; the extension compares sanitized session IDs against that directory index rather than issuing one filesystem IPC `stat` per SDK session. Its normal `shouldShowSession()` origin filter still applies afterward.

When Agent Host is enabled and configured as the default for the current surface, the extension-host service also skips its shared `~/.copilot/session-state/**/*.jsonl` watcher: `chat.agentHost.defaultSessionsProvider` controls the agent window, while `chat.defaultToCopilotHarness` controls editor windows. See [copilot-extension-host-cli § Shared SDK session catalog](./copilot-extension-host-cli.md#shared-sdk-session-catalog-ownership-and-refresh) for the complete ownership and refresh contract.

The suppression gate (`DefaultSessionsProviderContribution` checking `AgentHostEnabledSettingId` and early-returning before registering `CopilotChatSessionsProvider`) was removed — it is no longer needed. Both providers may advertise the same logical session-type id (`copilotcli`); the local provider returns the plain agent label from `_formatSessionTypeLabel` to avoid label conflicts (see the [gotcha below](#debt--gotchas)).

To visually distinguish local-agent-host sessions in the sidebar, workspace labels are formatted as `${folderName} [Local]` via `buildAgentHostSessionWorkspace`'s `providerLabel` parameter, matching the remote provider's `${folderName} [${hostName}]` pattern. The `[Local]` tag appears only on workspace labels, not session-type labels.

## One-shot `_ensureSessionCache` + auth-aware eager load

`BaseAgentHostSessionsProvider._ensureSessionCache()` runs `_refreshSessions()` — gated by a `_cacheInitialized` flag (and skipped while a call is already `_sessionRefreshInFlight` or a `_sessionRefreshRetry` is pending) — and is only invoked from `getSessions()`. `_cacheInitialized` flips to `true` **only when `listSessions()` actually returns** (even with an empty list); a call that *throws* does not initialize the cache. Once the cache is authoritative, further updates come from AHP notifications (`notify/sessionAdded`, `notify/sessionRemoved`, `notify/sessionSummaryChanged`).

This caching shape works only if the underlying agent answers `listSessions()` truthfully. A genuine empty **success** (`[]`) is authoritative and is cached as-is — the sidebar then stays empty until an AHP notification (typically the user sending a message → `notify/sessionAdded`) adds a session. A call that can't answer yet is expected to **throw** instead. See [the Copilot agent's authentication contract](./copilot-agent-provider.md#authentication-contract): the agent throws `AHP_AUTH_REQUIRED` rather than lying with `[]`. `_refreshSessions`'s blanket `catch` swallows that throw silently — intentional (we should NOT prompt the user to sign in just to render the sidebar) — and instead **arms a backoff retry** (`_scheduleSessionRefreshRetry`, `SESSION_REFRESH_RETRY_MIN_MS` 1s → `MAX_MS` 30s, cancelled on the next success), so a transient failed initial list self-heals without needing an unrelated AHP event.

The eager-load trigger is auth-aware, on the renderer side. `LocalAgentHostSessionsProvider`'s constructor registers `autorun(reader => { if (authenticationPending.read(reader)) return; this._cacheInitialized = true; this._refreshSessions(); })`. Because [`authenticationPending` is sticky](../changes/2026-04-17-session-loading-while-authenticating/summary.md) (settles to false exactly once), this fires `_refreshSessions()` precisely once: immediately if auth was already settled (reload case), or as soon as the first auth pass completes (fresh-launch case). The `_cacheInitialized = true` write before the call short-circuits any later `getSessions()` from re-firing.

The remote provider has its own equivalent: `setConnection(...)` explicitly calls `_refreshSessions()` after wiring listeners. Both providers converge on "refresh once we know we can get a real answer" — they just have different "we can get a real answer" signals.

This is the right layer for the fix. The alternative (changing `SessionsManagementService` to call `provider.getSessions()` after subscribing) would push knowledge of this provider's lazy-cache behavior into a generic consumer; agents are free to choose whether `getSessions()` triggers any work, and the provider should be self-sufficient about answering correctly when called.

## Startup session cache (cross-window persistence)

On top of the in-memory `_sessionCache`, `BaseAgentHostSessionsProvider` persists a lightweight snapshot of each session's summary to `IStorageService` and re-hydrates it on the next launch, so cached sessions render **immediately on window startup** — before the first live `listSessions()` round-trip (and, for local, before the agent host process has even finished spawning). This was originally remote-only and was hoisted into the base so the local provider gets it too (PR [#324328](https://github.com/microsoft/vscode/pull/324328)); it is behavior-preserving for remote and net-new for local.

Shape:

- **Opt-in, at the end of the subclass constructor.** A subclass calls `_enableSessionCachePersistence(storageKey)` to turn persistence on. It must be called **last**, once the identity fields hydration depends on are set — `_loadCachedSessions()` builds adapters via `createAdapter`, which reads `this.id`, `resourceSchemeForProvider(...)`, and `_adapterOptions()`. This is why the base constructor can't auto-load. Local uses a fixed key `localAgentHost.cachedSessions` (one machine-wide host); remote uses per-authority `remoteAgentHost.cachedSessions.${authority}`.
- **Serialization.** `ISerializedSessionMetadata` + `serializeMetadata`/`deserializeMetadata` store a URI-stringified subset (session, start/modified time, summary, workingDirectory, isRead, isArchived, project). Diffs/changesets are intentionally dropped — they re-populate on the next live refresh. Persisted under `StorageScope.APPLICATION` / `StorageTarget.USER`, capped at `CACHED_SESSIONS_MAX_PER_HOST = 100` most-recently-modified entries.
- **Dirty-flag write path.** `createAdapter`/`updateAdapter` snapshot the source `IAgentSessionMetadata` into `_metaByRawId` and set `_cacheDirty`; the actual write happens lazily on `IStorageService.onWillSaveState` (so notification bursts don't re-serialize the whole cache repeatedly). `_persistCache` overlays the mutable fields (`title`, `updatedAt`, `isRead`, `isArchived`) read from each adapter's observables onto its `_metaByRawId` snapshot.
- **Reconciliation.** Hydrated entries are provisional: the first successful `_refreshSessions()` prunes any that `listSessions()` no longer reports (see [One-shot `_ensureSessionCache`](#one-shot-_ensuresessioncache--auth-aware-eager-load)).
- **Offline gate (remote only).** `_shouldTrackSessionCacheChanges()` is a base hook (default `true`) the remote provider overrides to `!this._unpublished`, so while a host is marked unreachable (`unpublishCachedSessions()` hides the sessions from `getSessions()` but keeps the in-memory + on-disk cache) the snapshot is not clobbered and reappears on reconnect.

Startup-latency note: hydration makes the *persisted* list appear instantly, but the *live* list still waits on the agent host. For the **local** provider the persisted list can still appear later than the extension-host `CopilotChatSessionsProvider`'s list, because the latter reads `AgentSessionsModel`'s synchronously-loaded workspace cache (`agentSessions.model.cache`, `StorageScope.WORKSPACE`) while local hydration is tied to `LocalAgentHostContribution` construction at `WorkbenchPhase.AfterRestored` (idle-scheduled, behind the heavy `AgentHostContribution`). Registration → `SessionsManagementService.onDidChangeProviders` is event-driven, so earlier provider registration = earlier paint. Starting the agent host process earlier (prewarm) and reordering that contribution is tracked separately in [microsoft/vscode#324330](https://github.com/microsoft/vscode/issues/324330); see [agent-host-topology](./agent-host-topology.md#debt--gotchas) for the process-spawn timing.

## Session-type id vs. resource scheme

Two distinct concepts hang off a session type, and they diverged in the [2026-04-20 routing fix](../changes/2026-04-20-remote-agent-session-routing-fix/summary.md):

- **`ISession.sessionType` / `getSessionTypes()[i].id`** — the **logical session type** the new-chat picker uses to identify "which kind of session is this." This is now the agent's `provider` name itself, e.g. `copilotcli`. The same agent shares one logical session-type id across local and remote hosts, so a stale `sessionTypeId` from a previously-active local Copilot session does not blow up new-session creation when the user then picks a remote workspace exposing the same agent. This was the original fix for the misrouting bug.
- **Resource scheme** (the URI scheme on `ISession.resource`) — host-specific routing for the chat-session content provider. Local uses `agent-host-${provider}` (e.g. `agent-host-copilotcli`); remote uses `remoteAgentHostSessionTypeId(connectionAuthority, provider)` (e.g. `remote-ssh__macbook-air-copilotcli`). Computed by the abstract `protected resourceSchemeForProvider(provider)` hook.

The chat-sessions registry (`IChatSessionsService.registerChatSessionContentProvider`) is keyed by the **resource scheme** — `agent-host-copilotcli` for local, `remote-…-copilotcli` for remote. Each `(host × agent)` pair gets its own content provider, model provider, customization sync provider, etc. (See [agent-host-topology](./agent-host-topology.md#the-shared-seam-iagentconnection-and-agenthostsessionhandler).)

Inside the providers, `_syncSessionTypesFromRootState(rootState)` lives on the base and is shared across local and remote. It walks `rootState.agents` and exposes one session type per agent with `id = agent.provider`. There is no longer any alias-map indirection (`WELL_KNOWN_AGENT_SESSION_TYPES`, `_logicalSessionTypeForProvider`, `wellKnownAgentProvider`, `sessionTypeForProvider`, `agentProviderFromSessionType`) — what the agent advertises in `rootState.agents[].provider` is what the picker shows. The local provider previously had a `_getSessionTypesFromContributions()` fallback that derived session types from `IChatSessionsService` contributions when `rootState` hadn't hydrated; that's gone too — sessions types come from one source.

## New-session URI lifecycle

Agent Host session resources are owned by Agent Host providers/controllers, not by the generic chat-session fallback. With this PR, there should be no `agent-host-*:/untitled-*` resources in the Agent Host lifecycle after the generic chat widget has staged a blank contributed session. If one appears in an Agent Host provider/handler path, treat it as a bug.

The agent-window provider path works like this:

1. `BaseAgentHostSessionsProvider.createNewSession(workspaceUri, sessionTypeId)` chooses the agent provider (`sessionTypeId` is the logical agent provider id, e.g. `copilotcli`) and maps it to the chat resource scheme via `resourceSchemeForProvider(provider)`.
2. `_createNewSessionForType(...)` creates a `NewSession`: an `ISession` skeleton with `{ scheme: resourceScheme, path: '/<uuid>' }`, observable loading/status/model/config state, and the selected workspace/provider.
3. `NewSession.eagerCreate(connection)` calls `connection.createSession({ session: AgentSession.uri(provider, rawId), provider, workingDirectory, config })` immediately after the folder is picked, then opens a refcounted session-state subscription. The backend may mark the result `provisional`, meaning the in-memory AHP state exists but `notify/sessionAdded`, `SessionReady`, git-state attachment, and config persistence are deferred until the agent materializes the session on first send.
4. `AgentHostSessionHandler.provideChatSessionContent(resource)` is allowed to load this final-looking resource, but local/remote contribution wiring tells it the resource is still new: `IAgentHostSessionWorkingDirectoryResolver.registerResolver(resourceScheme, resolveWorkingDirectory, isNewSession)` returns true while `provider.getSessionByResource(resource)?.status` is `SessionStatus.Untitled`.
5. On first send, the handler peeks at the unmanaged subscription for the eagerly created backend session. If hydrated state exists, it skips the legacy create-then-subscribe path, dispatches the latest session config with `replace: true`, publishes the active client/tools/customizations, and runs the turn. If no eager state exists, it falls back to creating the backend session on first send.
6. When the backend materializes and reports the session through `listSessions()` / `notify/sessionAdded`, the provider graduates the `NewSession` into the running cache and releases the held subscription without disposing the backend. If the user switches folders or the send fails before graduation, disposing the `NewSession` releases the subscription and calls `connection.disposeSession(...)` so abandoned provisional sessions do not leak.

The provisional session may already have entered `_sessionCache` through an active-turn `listSessions()` overlay using the selected checkout. `notify/sessionAdded` is therefore an authoritative **upsert**, not create-only: `_handleSessionAdded` calls `AgentHostSessionAdapter.update` on an existing adapter, preserving identity while replacing project/working-directory data and firing `onDidChangeSessions.changed`. For remote providers, the shared update hook also refreshes `_metaByRawId` so the materialized worktree survives offline cache restoration.

Git metadata arrives separately after materialization. The host suppresses Git probes while lifecycle is `Creating`, then refreshes against the final worktree and publishes `_meta.git` via `SessionMetaChanged` plus `root/sessionSummaryChanged`. Both provider paths call `setMeta`, rebuild `workspace.repositories[0]`, and report a session change when user-visible Git fields change. See [agent-host-git-driven-diffs](./agent-host-git-driven-diffs.md#git-state-lifecycle-for-provisional-worktrees).

The workbench contributed-chat path is similar but starts one layer higher. `ChatServiceImpl.sendRequest` detects blank contributed chat sessions with `/untitled-*` internal resources and calls `IChatSessionsService.createNewChatSessionItem(...)` before dispatching the request. For local Agent Host, `AgentHostSessionListController.newChatSessionItem(...)` creates the final `agent-host-${provider}:/${uuid}` resource and remembers that raw id in `_pendingNewSessions`; `AgentHostContribution` passes `listController.isNewSession(resource)` into the handler. The returned item is a routing handle, not a visible list row: it is not inserted into `_items` until the backend session exists, preventing phantom rows if creation fails. `_pendingNewSessions` is cleared by `notify/sessionAdded`, `notify/sessionRemoved`, and `refresh()` when `listSessions()` observes the real session.

The remote sessions provider does not currently implement a local `newChatSessionItem` path for workbench chat, but its agent-window draft detection follows the same resolver-predicate model. Remote `createSession` must still honor a requested `config.session`; see [agent-host-remote-protocol-client](./agent-host-remote-protocol-client.md).



`BaseAgentHostSessionsProvider` (in `providers/agentHost/browser/baseAgentHostSessionsProvider.ts`) is `extends Disposable implements IAgentHostSessionsProvider`. It owns:

- All shared maps and observables: `_sessionCache`, `_pendingSession`, the active `NewSession` draft object, `_runningSessionConfigs`, `_sessionStateSubscriptions`, `_cacheInitialized`.
- All emitters: `_onDidChangeSessions`, `_onDidReplaceSession`, `_onDidChangeSessionConfig`.
- All connection-routed methods (`getSessions`, `getSessionConfig`, `setSessionConfigValue`, `archiveSession`, `sendRequest`, `sendAndCreateChat`, etc.) — they read the connection through `protected abstract get connection(): IAgentConnection | undefined` and skip dispatch (but still update local state) when undefined.
- All AHP notification/action handlers (`_handleSessionAdded`, `_handleConfigChanged`, etc.), wired up through `_attachConnectionListeners(connection, store)`. Local calls this once in the constructor with `this._store`; remote calls it from `setConnection` with a per-connection store so connection replacement disposes everything.
- A single concrete `AgentHostSessionAdapter` (one class, not subclassed) parameterised by an `IAgentHostAdapterOptions` bag: `{ icon, description, loading, buildWorkspace, mapDiffUri? }`. Session-type icons come from the agent descriptor when available, falling back to the provider/default icon.

Subclass hooks the base calls (defaults are local behaviour, remote overrides as needed):

- `protected abstract get connection()` — the live `IAgentConnection` or `undefined`.
- `protected abstract get authenticationPending: IObservable<boolean>` — adapter captures this on construction; remote's is sticky (see below).
- `protected abstract _adapterOptions(): Pick<IAgentHostAdapterOptions, 'description' | 'buildWorkspace'>` — the subclass-specific portion of the adapter options. Base's `createAdapter` (now concrete, not abstract) fills in `icon`, `loading`, and `mapDiffUri` from the corresponding hooks and merges in `_adapterOptions()`. Remote optionally overrides `createAdapter` itself to do extra bookkeeping (it stashes the metadata in `_metaByRawId`) and then delegates to `super.createAdapter()`.
- `protected abstract resourceSchemeForProvider(provider: string): string` — turns an agent provider name into the content-provider resource scheme. Local: `agent-host-${provider}`. Remote: `remoteAgentHostSessionTypeId(connectionAuthority, provider)`.
- `protected abstract _formatSessionTypeLabel(agentLabel: string): string` — formats the human-readable session-type label. The local provider returns the plain agent label (e.g. `Copilot`); the remote provider appends the host name (e.g. `Copilot [my-host]`). The local provider must NOT append `[Local]` because `SessionsManagementService._collectSessionTypes()` deduplicates by `type.id` (first-seen wins), and both the local provider and the extension-host `CopilotChatSessionsProvider` share the `copilotcli` type id — mismatched labels would be non-deterministic in the filter menu.
- `protected abstract resolveWorkspace(...)` — what folder to open with a session.
- `protected abstract id`, `label`, `icon`, `browseActions`.
- Optional overrides: `mapWorkingDirectoryUri`, `mapProjectUri`, `_diffUriMapper`, `onConnectionLost`, `_validateBeforeCreate`, `_noAgentsErrorMessage`, `_notConnectedSendErrorMessage`.

Note: `_syncSessionTypesFromRootState(rootState)` is now **on the base** (was duplicated across providers in different shapes). Both providers call it from their hydration path. See [Session-type id vs. resource scheme](#session-type-id-vs-resource-scheme).

**`update()` returns `boolean`.** Both refresh paths (`_refreshSessions` and the handlers) check the return of `adapter.update(metadata)` and only fire `onDidChangeSessions` when something actually changed. This was previously inconsistent across providers.

## Settings editor file-system providers

Next to the providers themselves, two sibling contributions expose AHP config as synthetic editable JSONC files in the workbench:

- `agentSessionSettings.contribution.ts` + `agentSessionSettingsFileSystemProvider.ts` register the `agent-session-settings://` scheme with `IFileService` and contribute an "Open Session Settings" action under `SessionItemContextMenuId`. Opens a per-session synthetic JSONC document backed by the session's resolved `SessionConfigState` (schema + values). Saves are written back through the provider's `setSessionConfigValue` (mid-session `SessionConfigChanged`).
- `agentHostSettings.contribution.ts` + `agentHostSettingsFileSystemProvider.ts` + `agentHostSettingsShared.ts` register the `agent-host-settings://` scheme and contribute an "Open Host Settings" action. Opens a host-level JSONC document backed by `RootState.config` (the new `RootConfigState`, see [agent-host-protocol](./agent-host-protocol.md)). Saves are written back via the host-level config dispatch path. The action is `f1: true` (Command Palette-enabled) but originally only worked from the session context menu, which passes an `ISession` argument; invoked from the palette there was no argument and the command silently no-op'd. PR #323061 fixed it to fall back to `ISessionsService.activeSession` when no context is passed, and added a precondition gating the palette entry on the active session being an agent-host provider.

Both contributions gate their menu `when` clauses on `ContextKeyExpr.regex(ChatSessionProviderIdContext.key, ANY_AGENT_HOST_PROVIDER_RE)` — the menu only shows for agent-host sessions.

On the **server** side, both per-session and host-level config values now read through `IAgentConfigurationService.getEffectiveValue<D, K>` (`src/vs/platform/agentHost/node/agentConfigurationService.ts`), which owns the `session → parent subagent → host` inheritance chain. Provider-side reads on the renderer still go through the lazy-seeded `_runningSessionConfigs` cache as described below; the new service is what they see *behind* the wire.

## Dynamic session config: the caches

The providers maintain related caches that together drive the session-config picker:

- `_newSessionConfigs: Map<sessionId, IResolveSessionConfigResult>` — for sessions being *created* in the welcome view. The picker writes values into this map as the user fills out the form.
- `_runningSessionConfigs: Map<sessionId, IResolveSessionConfigResult>` — for sessions that are already running. Drives the picker that lives in the chat input toolbar. Values are typed `Record<string, unknown>` (widened from `Record<string, string>` once the protocol started carrying number/boolean/object config values).
- `_sessionStateSubscriptions: DisposableMap<sessionId, IReference<IAgentSubscription<SessionState>>>` — lazy AHP session-state subscriptions, used to seed `_runningSessionConfigs` on demand.
- Plus host-level `RootConfigState` plumbing on the base for the host-settings editor (see [Settings editor file-system providers](#settings-editor-file-system-providers)).

`getSessionConfig(sessionId)` returns `_newSessionConfigs.get(sessionId) ?? _runningSessionConfigs.get(sessionId)` and synchronously kicks off `_ensureSessionStateSubscription(sessionId)` for the running case. The `??` ordering matters: the new-session form must always win over a stale running entry while the user is creating a session.

The picker side (`agentHostSessionConfigPicker.ts`) renders synchronously and tolerates `undefined` — it relies on `provider.onDidChangeSessionConfig` firing once a value arrives. The well-known `autoApprove` property is dispatched separately to a unified permission picker — see [agent-host-auto-approve-picker](./agent-host-auto-approve-picker.md).

## Lazy seeding from `SessionState.config`

Sessions opened from the cached list are not created in this window, so neither `_runningSessionConfigs` nor any incoming `SessionConfigChanged` action fires for them. The fix is the **lazy session-state subscription**: on first `getSessionConfig(sessionId)` for a known cached session that has no entry, the provider acquires `connection.getSubscription(StateComponents.Session, sessionUri)` (refcounted via `IReference`), seeds `_runningSessionConfigs` from `state.config` once it hydrates, and listens for subsequent `onDidChange` to keep its cache in sync.

Key invariants:

- **Refcounted subscription.** Both `IAgentHostService.getSubscription` and the remote `IAgentConnection.getSubscription` are refcounted. The Session Handler also holds the same wire subscription for any session whose chat content is loaded — the provider's seeding does not open a second wire subscription, it joins the existing one. Disposal of the provider's `IReference` is what releases the refcount; releasing the last one tears the wire subscription down.
- **Resolved-equality short-circuit.** `_seedRunningConfigFromState` compares the new candidate to the existing entry via `resolvedConfigsEqual` (exported from `src/vs/sessions/common/agentHostSessionsProvider.ts`). If structurally equal, it no-ops — this avoids spurious `onDidChangeSessionConfig` fires from re-seeds. Both providers import the helper from the shared module.
- **Lifecycle.** Local provider disposes the per-session subscription in `_handleSessionRemoved`. Remote provider stores the map under `_connectionListeners` so a connection replacement disposes every per-session subscription alongside the rest of the connection state.
- **Synchronous return contract preserved.** `getSessionConfig` still returns `undefined` on the first call for an unseeded session and lets the picker re-render when the seed arrives. The upstream picker `autorun` already handles that.

## Persistence: the bridge between AgentService and providers

Lazy seeding from `SessionState.config` would be enough on its own *if the server always rehydrated the full config schema and values on resume*. It doesn't always do that — config like `autoApprove` is stored per-window, and the agent side may have no record of what the user picked when the session was created. To paper over this, `AgentService` persists session config into the per-session Agent Host database, and the providers read it back through the normal session-state subscription path (the agent side reads it, includes it in `state.config.values`, the providers see it via the seed).

Two write sites in `src/vs/platform/agentHost/node/`:

- **`agentService.ts` `createSession`** persists the **full resolved values** (`sessionConfig.values`) on session create. The "full resolved values" choice is deliberate: clients render the resolved config on restore and shouldn't have to re-resolve it. The persisted values are read back as the source of truth for what the session was *actually created with*. They are **not** fed back into `resolveSessionConfig` as overrides on restore — restore returns them as-is.
- **`agentSideEffects.ts` `SessionConfigChanged`** persists `sessionState.config.values` verbatim each time the values change mid-session. Same rationale. The action now also carries an optional `replace?: boolean` — callers distinguish a full-values replacement (settings-editor save) from an incremental merge (single-property picker write); the persistence side stores whatever final shape the reducer produced.

Server-side **reads** (e.g. when an agent asks "what is the current value of `autoApprove` for this session?") flow through `IAgentConfigurationService.getEffectiveValue` (`src/vs/platform/agentHost/node/agentConfigurationService.ts`), which combines per-session values, parent subagent values, and host-level `RootConfigState` values into a single answer. The provider-side renderer cache is still seeded from `SessionState.config`, but the schema and value resolver now lives in the new service — reach for it (not the old per-call resolver) when adding a new well-known config key.

On restore (`restoreSession`), the agent reads the persisted values out of the database and includes them in the next `SessionState.config.values` snapshot. Both providers then see them through the lazy seed described above.

This was a previous reviewer suggestion to filter to user-mutable subsets only; we explicitly do **not** do that — see the gotcha entry below.

## Custom-agent selection: worktree relocation and resume hydration

`AgentHostSessionAdapter.mode` (the selected custom agent) needs to survive two situations the naive "just mirror the last-picked URI" model doesn't handle:

- **Worktree relocation** (PR #323548) — a session may graduate having picked an agent file in the original repo, then actually run in an isolated worktree, where the host reports the *same* agent file but rooted under the worktree path. `reconcileSelectedAgent(agents)` detects a stale selection (the current `mode.id` isn't in the host's current agent list), and — if the stale URI is inside the working directory it was picked under (`_agentBaseDir`, tracked alongside every `setChatAgent` call) — looks for a "twin" agent in the new list whose path ends with the same repo-relative suffix but is rooted elsewhere (`_findRelocatedAgent`). Deliberately keys off the **agent list**, not the session's reported working directory: worktree-pathed customizations arrive before either `SessionSummary` or `SessionState.workingDirectory` flips to the worktree path, so a working-directory-keyed rebase would miss the race and let the picker destructively reset the selection instead.
- **Resume hydration** (same PR) — a freshly (re)loaded adapter starts with `mode === undefined`, even though the user had picked an agent before reload. The selection is actually persisted on the **chat channel** (the default chat's `ChatState.draft.agent`), not the session channel, so `_hydrateAgentFromDraft` opens a (refcounted, shared with the session handler) subscription to the default chat's state and calls `AgentHostSessionAdapter.hydrateSelectedAgent(agentUri)` once a draft agent arrives. Hydration is one-shot and guarded to never override a live selection — it stops observing as soon as `mode` is set, whether by itself, by a concurrent new-session graduation seed, or by a user pick — so it can't leak or race a later selection.

## Surfacing session `_meta.git` to `workspace.repositories[0]`

Agent-host sessions need git state — branch name, base branch, ahead/behind counts, working-directory dirtiness, GitHub-remote-ness — to drive the changes-view tree-root label and gate the PR/merge buttons in the agent window's changes view. That state is computed **server-side** by `IAgentHostGitService` (in `src/vs/platform/agentHost/node/agentHostGitService.ts`, which shells out to `git`) and propagated to clients through the AHP `_meta` slot on `SessionState`:

1. `AgentService` uses `IAgentHostGitStateService` on create, restore, subscribe-without-meta, materialization, and turn completion. `AgentHostGitStateService.refreshSessionGitState` suppresses every request while lifecycle is `Creating`, because that session does not yet own its final working directory.
2. `refreshSessionGitState` calls `IAgentHostGitService.getSessionGitState(workingDirectory)`. If the result equals what's already in `state._meta.git` (compared with `equals` from `vs/base/common/objects`), it skips the action — turn-complete fires this on every turn, so dedup matters.
3. Otherwise it calls `agentHostStateManager.setSessionMeta(sessionKey, withSessionGitState(current, gitState))`, which dispatches a `SessionMetaChanged` action through the normal reducer path. Clients see it as part of `state._meta`.
4. On the workbench side, `AgentHostSessionAdapter.setMeta(meta)` (called by `_applySessionMetaFromState` whenever the subscription's `_meta` changes) reads it back via `readSessionGitState(meta)` and rebuilds the workspace through `buildAgentHostSessionWorkspace(...)` in `src/vs/sessions/common/agentHostSessionWorkspace.ts`, which fans the git fields out onto `workspace.repositories[0]` (`branchName`, `baseBranchName`, `baseBranchProtected`, `hasGitHubRemote`, `upstreamBranchName`, `incomingChanges`, `outgoingChanges`, `uncommittedChanges`).
5. `baseBranchProtected` is computed client-side from `baseBranchName` against `git.branchProtection` patterns — the server doesn't know the client's user settings, so the boolean is derived in `buildAgentHostSessionWorkspace` (`matchesAnyBranchProtectionPattern`).

The well-known shape lives in `src/vs/platform/agentHost/common/state/sessionState.ts` (`SESSION_META_GIT_KEY = 'git'`, `ISessionGitState`, `readSessionGitState`, `withSessionGitState`). The protocol-level `_meta` slot is generic — a `Record<string, unknown>` — so additional well-known keys can be added the same way without protocol churn.

The agent window's changes view consumes the resulting fields via `workspace.repositories[0]` in `changesViewModel.ts`. Note that the **editor-window session list** (`agentSessionsService.getSession(uri).metadata`, populated by `AgentHostSessionListController._buildMetadata`) does NOT currently carry these git fields — only `remoteAgentHost` and `workingDirectoryPath`. See the corresponding debt entry below.

## GitHub PR badges and the icon cache

Sessions whose working directory has an associated pull request show a PR badge in the list. The state that drives this rides one level up from `_meta.git`: it lives on `SessionSummary._meta` under `SESSION_META_GITHUB_KEY` (`'github'`), shape `ISessionGitHubState { owner; repo; pullRequestUrl }`, so the list APIs can render a badge without subscribing to full `SessionState`. Server-side, `IAgentHostGitStateService` computes it (`refreshSessionGitState`, `setSessionGitHubState`, `attachSessionGitHubPullRequest`); the underlying GitHub/Octokit access goes through `IAgentHostOctoKitService` and PR title/description generation through `ICopilotApiService`.

On the renderer, the agent window's GitHub surface lives under `src/vs/sessions/contrib/github/`:

- `githubService` — resolves PR metadata for a session.
- `pullRequestIconCache` — a **persistent** cache (storage key `sessions.github.pullRequestIconCache`, capped at `MAX = 50` entries) so the PR badge renders immediately on reload instead of flashing while the network resolves.
- `pullRequestIconStatus` — the open/merged/closed status that selects the icon.

`providers/agentHost/browser/sessionGitHubInfo.ts` adapts the agent-host session's `_meta.github` into the shape this surface consumes. The PR / commit / sync / discard *operations* themselves are changeset operations — see [agent-host-git-driven-diffs](./agent-host-git-driven-diffs.md#changeset-channel--service-decomposition).

## Changeset restore, branch-changes diff summary, and Session Files

Several commits since the 2026-06-25 baseline refined how the provider surfaces changes-view data for the sidebar and Changes view, on top of the [changeset channel](./agent-host-git-driven-diffs.md#changeset-channel--service-decomposition):

- **Changesets restore on window reload** (PR #323494) — `AgentHostSessionAdapter` now seeds its changeset catalogue directly from the `SessionState` snapshot's `changesets` field on initial hydration (`_applyChangesetsFromState`), rather than waiting for a live `SessionChangesetsChanged` action that the host only emits on subsequent changes. Without this, reloading the window showed an empty Changes view until the next changeset mutation.
- **Session list badge / title now use the branch-changes diff summary** (PR #323852) — the adapter subscribes to the `Branch Changes` changeset (`buildBranchChangesetUri`, the branch-scoped sibling of `buildSessionChangesetUri`) via `StateComponents.Changeset` and derives `changesSummary`/session-title decoration from that changeset's live diff summary, replacing an earlier ad hoc computation. This subscription and its downstream derived observables were substantially simplified in a follow-up cleanup pass (PR #323887, ~100 net LOC removed) and had a loading-state edge case fixed (PR #323735) and flicker reduced when the change list/actions re-render (PR #323593).
- **Session Files section** (PR #323725) — a new optional `externalChanges` observable on `ISession`, populated by `createSessionFilesObs` (`agentHostSessionFiles.ts`), surfaces files the agent created/edited/deleted **outside** the session's workspace folders during the session (never committed, so they don't show up in the git-diff-driven Changes view). Parsed incrementally from agent-host chat-state turns (response parts → tool calls → file-edit results/pending edits), memoized per completed turn so only the active turn is re-parsed per streamed delta. Backs a new collapsible/resizable "Session Files" section in the Changes view, modeled on the CI Checks section.
- **PR badge shows immediately on detection** (PR #323740) — the open-PR icon now renders as soon as a PR is detected rather than waiting for the richer status (open/merged/closed) to resolve, reducing perceived latency; the persistent `pullRequestIconCache` (see [GitHub PR badges and the icon cache](#github-pr-badges-and-the-icon-cache)) still backs the cross-reload fast path.

## Agent SDK download progress

Claude and Codex ship without their native SDK binaries; those are downloaded on demand from a CDN the first time a session of that provider materializes, then cached on disk (70-95 MB tarballs — a cold-cache first session start used to block for several seconds with no feedback). `NewSession.eagerCreate` now supplies a `progressToken` (a fresh UUID) on its eager `createSession` call, MCP-style opt-in for the host to report progress against; the host echoes the token on generic `root/progress` notifications (`ProgressParams`) as the download proceeds, terminating with a frame where `progress === total`. `BaseAgentHostSessionsProvider` tracks one `IActiveDownload` per in-flight token (`_activeDownloads`), driving an `IProgressService` notification (`ProgressLocation.Notification`) that reports determinate percentage steps and dismisses itself on completion. `isSdkResolvableWithoutDownload` / `canLoadWithoutDownload` guard eager/background callers (e.g. `listSessions()` at startup) so they never accidentally trigger a cold download just by enumerating sessions.

## Workspace trust

Eager backend session creation (`NewSession.eagerCreate`, the one-shot cache load) is gated on workspace trust. `BaseAgentHostSessionsProvider` carries `requiresWorkspaceTrust` and a `getUriTrustInfo(uri)` helper; for an untrusted folder it **skips the eager `createSession`** so an agent doesn't start running against code the user hasn't trusted. The session is created normally once trust is granted (or once the user explicitly proceeds).

## Quick chats (workspace-less sessions)

The agent window now supports **quick chats**: lightweight, single-chat agent-host sessions that aren't scoped to any workspace folder. They render in an always-visible in-list "Chats" section, persist across reloads, and use the normal session presentation (with the "Done" affordance hidden, since there's no workspace to finish reviewing).

- **`IAgentHostSessionKind` strategy.** `BaseAgentHostSessionsProvider` captures the workspace-vs-quick-chat distinction once, at construction, as one of two singleton strategy objects — `WorkspaceSessionKind` (`requiresWorkspace: true`, untitled title "New Session") or `QuickChatSessionKind` (`requiresWorkspace: false`, untitled title "New Chat", `computeWorkspace` always returns `undefined`). Both `AgentHostSessionAdapter` and the draft `NewSession` class hold a `_kind: IAgentHostSessionKind` chosen from `sessionKind(isQuickChat)` and delegate to it (`_computeWorkspace()`, `untitledTitle`, capability gating) instead of re-branching on the workspace-less tag throughout the class.
- **Provider capability.** `ISessionsProvider` gained `supportsQuickChats: boolean` and a **mandatory** `createQuickChat(sessionTypeId): ISession` (previously this would have been an optional method probed for at call sites — the mandatory form throws for providers that don't support it, so callers gate solely on the capability flag). `BaseAgentHostSessionsProvider.createQuickChat` resolves the session type and calls the same `_createDraftSession(sessionType, workspace, quickChat)` helper `createNewSession` uses, just with `workspace: undefined` and `quickChat: true` — a quick chat is otherwise the same session type, run by the agent host in a throwaway scratch working directory it assigns itself.
- **Workspace-less inference and tagging.** The agent host infers "workspace-less" from an absent `workingDirectory` on `createSession` (forks are excluded — a fork always has a working directory). The result is tagged onto the session via a generic `_meta.workspaceless` bag entry, read back with `readSessionWorkspaceless(meta)` (`sessionState.ts`). `AgentHostSessionAdapter` carries this forward as `_meta` on its cached-session construction path specifically so the session-kind resolves correctly the moment the adapter is built — the kind is fixed once and does **not** flip on a later `update()`/`setMeta()` call.
- **Ownership centralized in `AgentService`.** Each agent used to persist and re-emit its own namespaced workspace-less marker (`copilot.workspaceless`, `claude.workspaceless`), and an agent that didn't persist one (Codex) lost the marker on restart. `AgentService` is now the single owner: it persists one `agentHost.workspaceless` key at session create/materialize time (from the value it already infers), overlays `_meta.workspaceless` onto every agent's summary in `listSessions()`, and folds the same key into the restored summary `_meta` in `restoreSession()`. Per-agent re-emission was removed — Copilot still *reads* the AH-owned key (for its resume system prompt and scratch-dir cleanup) but no longer writes or re-projects it, and Claude's now-dead per-session workspace-less plumbing was deleted outright. This is what makes restored quick chats work for every agent, including ones with no dedicated workspace-less code of their own.
- **Capability gating.** `AgentHostSessionAdapter.capabilities.supportsMultipleChats` is forced `false` for a quick-chat session regardless of what the agent's `AgentCapabilities.multipleChats` advertises — quick chats are single-chat by design (no fork, no additional peer chats).
- **Internal naming.** Inside the agent-host layer the concept is named "workspace-less chat" (`COPILOT_AGENT_HOST_WORKSPACELESS_INSTRUCTIONS`, the `<workspaceless_chat>` system-message tag, `_quickChatScratchDir` internal helper names were intentionally kept as-is) — "Quick Chat" is kept only as the user-facing workbench term, and the agent-host prose documents that mapping rather than renaming everything to match the UI term.



## Multi-chat in the provider

With [multi-chat sessions](./agent-host-protocol.md#multi-chat-sessions), a single `ISession` can contain several chats. The provider exposes "new chat" / "fork" affordances that issue `createChat` (optionally with a `ChatForkSource`) against the session's connection, and the adapter aggregates the chat catalog into the single `SessionSummary` the list row renders (status/activity/`modifiedAt` rolled up per the default-chat compat rules). Per-chat overrides (`model` / `agent`) are not aggregated up to the session row. Multi-chat support extended to the **Claude** agent-host harness (PR #323625, matching the existing Copilot CLI experience) — the provider now advertises `supportsMultipleChats` for the `claude` logical session type as well as `copilotcli`; the corresponding per-chat persistence and fork/routing logic lives entirely in the Claude agent backend, out of this doc's scope.

Progress and changes now aggregate correctly across peer chats (PR #323089, fixes [#322418](https://github.com/microsoft/vscode/issues/322418)): previously a session's status stayed `Idle` (and its "Changes" pill never appeared) while a peer chat was actually running, because both were computed from the default chat/session DB alone. `AgentHostSessionAdapter._defaultChatStatusOverride` decouples the default chat's tab status from the session-aggregate status once `supportsMultipleChats && state.chats.length > 1` — in the single-chat case the override is cleared and the default chat again just mirrors the session status directly. The session-level changeset union (peer-chat DBs unioned into the session DB) is host-side (`sessionDiffAggregator.ts`, `agentHostChangesetService.ts`) and always does a full recompute for multi-chat sessions rather than the single-chat incremental fast/slow paths — a documented follow-up, not yet incremental.

**Read-only subagent chats as peer tabs** (PR #323960). `IChat` gained `interactivity: IObservable<ChatInteractivity>` (`Full | ReadOnly | Hidden`), mapped from the protocol's `ChatInteractivity` via `toChatInteractivity` (defaults to `Full` for backward compatibility when the field is absent). `applyChatCatalog`'s peer-surfacing rule changed from "surface every non-default chat when the session supports multiple chats" to `surfacesAsPeer(summary)`: a non-default chat surfaces as a peer tab when the session supports multiple chats **OR** the chat's origin is `ProtocolChatOriginKind.Tool` (a subagent/worker chat) — independent of multi-chat support. Subagent peer chats are always read-only. `_resolveParentChatResource` maps a Tool/Fork origin's protocol parent-chat URI back to this session's UI chat resource (default chat vs. a peer chat's fragment-encoded id), which is what backs the transcript's inline "Open Subagent" pill and the "Subagents" dropdown.

**Hide subagent chats by default; persist closed chats** (PR #324021). Subagent (tool-origin) chats are hidden from the tab strip by default and only appear as a tab once explicitly opened (e.g. from the Subagents dropdown); closing one just hides it again — it is deliberately excluded from the persisted "closed chats" set (subagent visibility always reverts to hidden-by-default on reload). For ordinary chats, closing is now remembered across reload/restart via a per-session persisted closed-chat set. `IChat` also gained per-chat `IChatCapabilities` (`canRename` / `canDelete`), resolved centrally (folding in the invariant that the main/default chat is never deletable) rather than via ad hoc origin/main-chat checks scattered across the tab context menu, delete keybinding, and context keys.

**Peer chats stay loaded across the first send.** A newly-created peer chat (opened via the lightweight new-chat composer, not a `ChatWidget`) has no widget holding its `IChatModel` reference. `BaseAgentHostSessionsProvider.sendRequest` for an already-committed session now explicitly loads and **holds** the target chat model through `IChatService` for the duration of the first send, applies the cached model/agent input state before dispatching, and releases the reference once the request settles. Without this, `ChatService` could unload the model mid-send (nothing else was keeping it alive) and reject the in-flight request as belonging to an unknown session.

**Archived sessions are read-only (banner + Restore).** Archiving a session is now surfaced as read-only across every chat in it, and this is a **derivation**, not a stored flag. The shared helper `effectiveChatInteractivity(isArchived, interactivity)` (`vs/sessions/services/sessions/common/session.ts`) folds the session's `isArchived` observable into each chat's effective interactivity: `Hidden` stays `Hidden`; otherwise archived downgrades to `ReadOnly`; otherwise the chat keeps its own interactivity. Both the default chat and each `AdditionalChat` derive their `IChat.interactivity` through this helper (the `AdditionalChat` constructor takes the session's `isArchived` observable, alongside the `lastTurnChanges` param). The agent-window `SessionView` (`vs/sessions/browser/parts/sessionView.ts`) shows a `SessionReadOnlyBanner` in place of the composer for any read-only chat; for an archived session it renders an archived-specific message plus an inline **Restore** action that runs `UNARCHIVE_SESSION_COMMAND_ID` (`vs/sessions/common/sessionCommands.ts` — extracted so both the action registration and the banner can share it without crossing the `parts`/`contrib` layering). Host-side enforcement (rejecting turns to a read-only/archived chat) is agent-agnostic and lives in `AgentSideEffects`; see [copilot-agent-provider § resume working-directory repair](./copilot-agent-provider.md#resume-working-directory-repair). The transcript still renders read-only even when the underlying worktree is gone (the repair path resumes archived sessions against the repo root for history).

## Tunnel-backed remote providers

Tunnel-backed remotes are surfaced by `TunnelAgentHostContribution`, not by the generic `RemoteAgentHostContribution`. The tunnel service owns the cached tunnel list and the persisted auto-connect suppression list:

- `getCachedTunnels()` is the recently used / discoverable tunnel cache.
- `suppressAutoConnect(tunnelId)` records that the user explicitly disconnected a tunnel.
- `clearAutoConnectSuppression(tunnelId)` clears that intent when the user explicitly reconnects or picks the tunnel again.

The contribution filters suppressed tunnels out of provider reconciliation. A user-disconnected tunnel should not register a `RemoteAgentHostSessionsProvider`, should not appear in the session workspace picker's Remote tab as "Connecting" or "Offline", and should not participate in wake/status-check auto-reconnect loops. The tunnel picker can still discover the tunnel and `cacheTunnel(...)` clears suppression after the user explicitly chooses it.

Background auto-connect has two suppression checks: one before starting, and one immediately after `await tunnelService.connect(...)`. The second check is load-bearing because the user can click the X while a startup/background connect is already in flight; if suppression appears during that await, the background connection must be disconnected again and must not clear the user's intent.

User-initiated reconnect clears suppression at the start of the attempt, not only after success. If the explicit attempt fails transiently, the normal retry loop should be allowed to continue instead of being blocked by the previous disconnect marker.

## SSH-backed remote providers: disconnect intent is removal

`RemoteAgentHostContribution` wires SSH-backed remotes through the same `provider.disconnect()` → `_disconnectOnDemand` hook that tunnel-backed providers use, but the meaning of "user disconnected" is different:

- **Tunnel** — disconnect intent is persisted as a suppression marker; the cached tunnel stays in storage and the tunnel picker can rediscover it (see above). The entry itself stays around because the cached tunnel metadata is valuable.
- **SSH** — there is no suppression marker. An SSH `IRemoteAgentHostSSHConnection` entry is essentially `{name, sshConfigHost}`; the real configuration lives in `~/.ssh/config`. Disconnect intent is simply *removal* of the configured entry. The user can re-add by re-picking the alias from the SSH picker.

`_disconnectSSHOnDemand` therefore calls `removeRemoteAgentHost(connection.address)` **before** `_sshService.disconnect(connectionKey)`. Ordering is load-bearing: `_sshService.disconnect` fires `onDidCloseConnection` → `notifyConnectionClosed` → `onDidChangeConnections` synchronously during its await, which the contribution translates into `_reconcile` → `_reconnectSSHEntries`. If the entry is still in `RemoteAgentHostsSettingId` storage at that point, the auto-reconnect path immediately reconnects the host we just told it to disconnect (and the persisted entry also auto-connects on the next reload).

`removeRemoteAgentHost` itself disposes the entry's `transportDisposable`, which for SSH calls `_mainService.disconnect(connectionId)` to tear down the main-process SSH tunnel (see [agent-host-topology](./agent-host-topology.md#debt--gotchas) for the per-entry `DisposableStore` ownership boundary established by the 2026-04-22 fix). The explicit `_sshService.disconnect(connectionKey)` afterwards is belt-and-suspenders to clear the connection by its connection-key as well.

This is why the [remote-host management UX](./remote-agent-host-management-ux.md#session-workspace-picker-inline-remove) doc says the inline X must use the shared `removeRemoteHost(...)` helper that calls `provider.disconnect()` rather than `IRemoteAgentHostService.removeRemoteAgentHost(address)` directly: each provider's `_disconnectOnDemand` hook decides what "user-initiated disconnect" means for that transport (tunnel: persist suppression; SSH: remove entry).

The ordering and the helper-call shape are exercised by `disconnectSSHEntry` in `remoteAgentHost.contribution.ts` and the corresponding `disconnectSSHEntry` / `sshConnectionKey` tests in `remoteAgentHost.contribution.test.ts`.

- **gotcha** (2026-04-22, sessionDatabase.ts:setMetadata) — `setMetadata` writes are sequenced **per key** via `SequencerByKey<string>`. `@vscode/sqlite3` runs in *parallelized* mode by default: two `db.run()` calls on the same connection are dispatched to libuv's thread pool and can complete out of submission order. For "last-writer-wins" keys like `configValues` (written by both `agentService.createSession` and `agentSideEffects.SessionConfigChanged`), unsequenced writes let the older value silently overwrite the newer one on disk. Surfaced as flaky `sessionConfig.integrationTest.ts` failures (PR [#311989](https://github.com/microsoft/vscode/pull/311989)). See the learning recorded in `.github/instructions/agentHostTesting.instructions.md` and `changes/2026-04-23-fix-undefined-branch-name/`.
