# Agent Host terminal-tool bug bash + launch-skill hardening

**Date:** 2026-05-26
**VS Code branch:** agents/vsckb-explore-i-d-like-you-to-do-14f3e7c5
**VS Code SHA at finalize:** e6e488e018
**PR:** https://github.com/microsoft/vscode/pull/318488 (draft — launch-skill changes only; the bug-bash findings themselves are not fixed in this PR)

## What was done

Ran a 10-scenario bug bash on the **Local Agent Host terminal tool** in the Agents window, with `chat.agentHost.customTerminalTool.enabled` left at its default of `false` (i.e. exercising the SDK's default terminal behavior). Each scenario was driven by a fresh subagent that launched its own Code OSS instance, created a new session in a pre-seeded `/tmp` workspace, sent a targeted prompt, and recorded screenshots + logs + a JSON report.

Findings — 12 deduped issues across the terminal-tool surface, organized as `files/bug-bash/{REPORT.md,issues.json,s1…s10/}` on the PR branch:

- **High severity (2):** session restore across Code OSS restart is broken — freshly-created sessions disappear from the sidebar even when the relaunch reuses the same `--user-data-dir`, `--extensions-dir`, and `--shared-data-dir`; pre-existing sessions remain visible but go read-only ("Sorry, no response was returned" on every new prompt).
- **Medium severity (3):** terminal tool cards disappear from the chat transcript after a multi-command turn completes; in-flight rows during such turns are too minimal (no chevron / output preview); cancelling a long-running command via `Cancel (⌘Escape)` ends the turn but the resulting card text still reads "Executed …" with no cancelled / stopped state.
- **Low severity (7):** stale "Running …" header after cancel; no toast / inline confirmation when cancel succeeds; no explicit exit-code badge on the tool card UI (failures use a red-dot icon + agent prose only); approval prompt renders simultaneously in two places (sidebar entry + bottom overlay); ANSI color escapes are silently stripped from terminal output; Files Explorer in Session Details fails with `ENOPRO: No file system provider found` for `/tmp/...` workspaces; leaked disposables logged from `SessionView._handleContextKeys` and `TitleBarAccountWidget.refreshAvatar`.

Positive results: long output (`seq 1 500`) renders cleanly with an internal xterm-style scroll region; unicode (emoji + CJK + accented + Greek) renders correctly across writing systems; failure rendering (stderr in a dark box, red dot icon) is clear; no state leakage when creating multiple fresh sessions in different workspaces back-to-back; same-launch session reopen via the sidebar restores the transcript and tool cards fully intact.

The bug bash also surfaced **three latent gaps in the `launch` skill itself** that the PR fixes:

1. `launch.sh` now sets `files.simpleDialog.enable: true` in the launched profile's `User/settings.json`. Without this, the new-session workspace picker's only action is `Select…` → native OS file dialog, which is unreachable over CDP/SSH.
2. New `scripts/monaco-paste.sh` helper inserts text into the focused chat-input Monaco editor by dispatching a synthetic `ClipboardEvent('paste')` with a `DataTransfer` payload. Avoids `pbcopy`'s system-wide `NSPasteboard` collision, handles unicode/emoji/backticks/newlines, normalizes Monaco's NBSP-for-space rendering, and waits two `requestAnimationFrame`s before read-back (Monaco updates view-lines asynchronously after paste).
3. `SKILL.md` rewritten so every `npx @playwright/cli` example now defaults to `-s=$PW_SESSION`. Without an explicit `--session`, parallel callers all share the implicit `"default"` daemon and the most-recently-attached CDP wins for every subsequent command from any of those shells. New "Parallel multi-instance pattern" subsection makes the convention explicit.

## Key decisions

- **Test infra over fixes, at least in this PR.** The bug bash uncovered 12 user-facing issues, but the PR scope deliberately stays at launch-skill improvements — the dev environment needs to be solid before any of those bugs can be fixed reliably. The bugs themselves are recorded as `debt:` entries on the relevant component docs and enumerated in this summary, to be picked up later.
- **Don't rely on `PLAYWRIGHT_CLI_SESSION` env var with `attach --cdp=`.** The env var is documented in the package README for `open`-style workflows, but `playwright-core/src/tools/cli-client/session.ts:142-143` also pushes its value as `--endpoint=` whenever `--cdp=` is the only target, causing the daemon to `ENOENT` on what it thinks is a socket path. Explicit `-s=NAME` works in all modes. (Worth filing upstream — see the line range above.)
- **Sequential parallelism for bug-bash subagents on macOS.** Beyond ~2–3 concurrent Code OSS instances, Crashpad's exception handler dies in a `mach_port_request_notification: invalid capability` loop. Not a `launch.sh` bug; OS-level Mach-ports exhaustion. Tooling now documents the limit; the bug bash itself was re-run sequentially after the initial 4-way batch corrupted three of four runs.
- **JSONC settings merge is text-based, not parser-based.** First implementation regex-stripped `//` and `/* */` comments then `JSON.parse`'d the result, with `{}` as the fallback on parse failure. That trashed user comments and silently dropped settings on URLs (`http://...` inside string values had its `//` stripped, breaking parse). The reviewed version finds the last `}` in the file, inserts the new key just before it, and fails loudly on truly-broken input rather than overwriting. Idempotent for already-true values; updates an existing false/string/etc. value via a targeted regex on the value slot only.

## What went wrong or was misunderstood

- **Tried 4-way parallel subagents on macOS without checking the resource ceiling.** Hit Crashpad `mach_port_request_notification: (os/kern) invalid capability` on three of four. Throttled to sequential and resumed. — **prevented by:** new `testing.md` § "Exploratory UI testing via the launch skill" entry on the macOS Mach-ports cap, plus a stronger note in `launch/SKILL.md`.
- **Attributed S3's cross-contamination to `pbcopy` clipboard collision alone.** In reality the dominant factor was `@playwright/cli`'s implicit `"default"` session collision — all four parallel subagents were running un-flagged `attach --cdp=...` calls and the most-recently-attached CDP won for every subsequent command from any of them. The user pushed on the diagnosis and I reassessed. `pbcopy` was at most secondary. — **prevented by:** `launch/SKILL.md` now defaults every example to `-s=$PW_SESSION`; `pbcopy` is now option 3 with the system-wide-pasteboard warning called out explicitly; `testing.md` § "Exploratory UI testing" lists this as one of three coordination gotchas.
- **Overclaimed `PLAYWRIGHT_CLI_SESSION` as "a bug in this alpha CLI"** without checking the docs first. It's actually documented (for `open` workflows) — it just interacts poorly with `attach --cdp=`. The user asked "is that documented?" and I had to walk it back. — **prevented by:** `launch/SKILL.md` now describes the env var accurately ("works for `open`-style workflows, interacts poorly with `attach --cdp=`") and points the curious at the line numbers in `playwright-core/src/tools/cli-client/session.ts`.
- **First settings-merge implementation would have dropped user data.** Naive regex-strip-then-`JSON.parse` with `{}` fallback would have eaten comments and silently overwritten on any parse error (e.g. a URL containing `//` inside a string). PR review caught both. — **prevented by:** rewritten as a text-based insert that preserves comments + string contents, fails loudly on structurally-broken input, and is idempotent. Seven test cases in the commit message including JSONC + URLs + malformed input.
- **`PW_ARGS[@]` under `set -u` is an unbound-variable error when empty.** Tripped the helper when no `--session` / `$PW_SESSION` was set. Spotted in post-commit verification, fixed with the `${PW_ARGS[@]+"${PW_ARGS[@]}"}` idiom at every call site. — **prevented by:** test pass in the PR-review-fix commit now covers the no-session path.
- **The new-session workspace picker doesn't work over CDP/SSH out of the box.** The slim-copied `--user-data-dir` carries through user settings, but the per-launch `--shared-data-dir` is empty, so `history.recentlyOpenedPathsList` has no entries and the picker's only action is `Select…` → native OS dialog. I was about to write a "seed the shared-data-dir from the source" branch when the user pointed out that `files.simpleDialog.enable=true` makes `Select…` open VS Code's quick-input-style folder browser instead, which IS automatable. Much smaller fix. — **prevented by:** `launch.sh` now sets this automatically.
- **The bug-bash subagent prompt assumed sessions appear under a "Local Agent Host" sidebar group.** They actually appear under a group label that's the workspace folder name uppercased (`BUG-BASH-WS`, `BUG-BASH-WS-S2`, etc.). Wasted a few minutes of the first subagent's run hunting for the wrong label. — **prevented by:** later subagent prompts were corrected; not a doc-level concern, but recorded here so the next "drive the Agents window from a subagent" attempt knows.

The retrospective is intentionally heavy on workflow / tooling misses rather than agent-host architecture surprises — the agent-host docs themselves were accurate where this session intersected them. The misses were mostly in the dev environment around the agent host.

## What we learned

- **The `launch` skill is mature enough to be a real test surface, not just a debugging convenience.** A 10-scenario bug bash through it produced 12 actionable findings and required only modest tool improvements to be repeatable. The four-layer testing model in `testing.md` is the right home for automated tests, but exploratory UI testing has a legitimate place above it.
- **`@playwright/cli` is alpha and has rough edges.** Daemon naming via `-s=` works; env-var override doesn't compose with `attach --cdp=`; multi-instance attach state is process-global (per-session-name), not per-cwd. The skill docs now reflect this honestly.
- **Per-instance isolation extends through the automation layer or it doesn't extend at all.** The launch skill carefully isolates per-launch UDD / shared-data-dir / ports; that work is wasted if every shell driving those instances shares one Playwright daemon and one OS pasteboard. The `-s=$PW_SESSION` discipline and the synthetic-paste recipe close that gap.
