# Persist agent-host workspace selection across reload

**Date:** 2026-04-24
**VS Code branch:** roblou/agents/persist-agent-host-selection
**VS Code SHA at finalize:** 99e71fd463b
**PR:** [#312037](https://github.com/microsoft/vscode/pull/312037)

## What was done

When the user picked a workspace folder hosted on an agent-host provider (local or remote/SSH/tunnel) on the new-chat page and reloaded the window, the Agents app was forgetting the selection and silently falling back to a different local folder. Two distinct races contributed:

1. **Provider registration order.** `LocalAgentHostSessionsProvider`, `CopilotChatSessionsProvider`, and remote/tunnel providers all register at `WorkbenchPhase.AfterRestored`, in non-deterministic order. `SessionWorkspacePicker._restoreSelectedWorkspace` runs eagerly during construction and filters out any stored entry whose `providerId` isn't registered yet — so an agent-host pick saved across reload would be discarded if its provider hadn't registered when the picker first looked.
2. **Remote connection lifecycle.** Remote providers (`RemoteAgentHostSessionsProvider`) always initialize `connectionStatus` as `Disconnected` and only flip to `Connecting`/`Connected` inside `tunnelAgentHost.contribution.ts`'s `_createProvider` — which calls `setConnectionStatus(Connecting)` *before* `registerProvider()`, both synchronously. By the time external code subscribes via `onDidChangeProviders`, the transitions may have already raced past. Restoring an unreachable SSH/tunnel selection therefore left the picker visibly stuck on the bad workspace with the wrong session attached, with no observable status change to react to.

The fix lives entirely in `src/vs/sessions/contrib/chat/browser/sessionWorkspacePicker.ts`:

- **Re-restore on `onDidChangeProviders`.** A `_userHasPicked` gate distinguishes "we auto-restored a fallback" from "the user explicitly picked something". While the gate is false, late provider registrations re-run `_restoreSelectedWorkspace` so the picker upgrades from any fallback to the user's actual stored "checked" entry once its provider arrives. Once the user has acted (`_selectProject` / `clearSelection`), providers coming and going never move the selection out from under them.
- **Validate-on-removal of selected provider.** When the currently selected workspace's provider disappears, the picker clears `_selectedWorkspace`, clears the connection-status watch, and fires `onDidChangeSelection` + `onDidSelectWorkspace(undefined)` so `NewChatViewPane._onWorkspaceSelected` calls `unsetNewSession()` and the UI doesn't show a stale session.
- **Connection-failure fallback for restored remotes.** A `_watchForConnectionFailure` helper is started after the initial restore and after any `onDidChangeProviders` upgrade. It owns a `MutableDisposable<DisposableStore>` containing both an `autorun` (clears on `Connected`, falls back on a *post-first-run* `Disconnected`) and a `disposableTimeout(..., RESTORE_CONNECT_GRACE_MS = 5_000)` safety net for the case where the provider's status flips before the autorun even subscribes (so we never observe a transition). On fallback, the picker fires `onDidSelectWorkspace(undefined)` — the same chain that drives `unsetNewSession()`. Both watches are cleared the moment the user picks anything explicitly.
- The previous `NewChatViewPane._createNewSession` workaround that watched the agent-host provider's connection status from the view-pane side was removed; the picker is the right owner because it already drives session creation/teardown via `onDidSelectWorkspace`.
- 13 unit tests in `sessionWorkspacePicker.test.ts`, including 4 grace-period tests using `runWithFakedTimers`.

## Key decisions

- **Picker owns the fallback, not the view pane.** First attempt put a `Disconnected`-aware autorun in `NewChatViewPane._createNewSession`. Moved it into the picker because the picker already owns the selection lifecycle and the view pane's `onDidSelectWorkspace(undefined)` handler already does the right thing (`unsetNewSession()` + clear pending session-type wait). This kept the view pane simple and put the policy where the state lives.
- **Grace-period timer instead of "wait forever".** Considered detecting "this provider id is never going to register" by inspecting registered contribution metadata, but that would have required new infrastructure. A 5-second `disposableTimeout` safety net combined with a `_userHasPicked` gate covers the SSH-can't-connect case without complicating the registration model.
- **`isFirstRun` guard on the autorun.** `autorun` runs synchronously on creation, so the first `read()` always sees the current `Disconnected` value. We skip fallback on the initial read, then fall back on subsequent `Disconnected` values. The grace-period timer is the safety net for the case where transitions race ahead of the subscription entirely.
- **Use `disposableTimeout(..., store)`, not raw `setTimeout` + `toDisposable(clearTimeout)`.** Equivalent behaviour, idiomatic.
- **`onDidSelectWorkspace` (action signal) vs `onDidChangeSelection` (render signal) are distinct.** The former drives session lifecycle (create/unset); the latter is fire-and-forget render notification. Selection-clearing paths need to fire both.
- **Strip the offline-styling work.** An earlier iteration added spinner / grayed-out trigger styling and a `_connectionStatusFor` watcher. Upstream PR [#312063](https://github.com/microsoft/vscode/pull/312063) (`2026-04-22-gray-out-remembered-folders`) already covers the styling, so this PR was simplified down to just the persistence fix.

## What went wrong or was misunderstood

- **Wrong initial mental model of remote provider lifecycle.** Assumed `Disconnected → Connecting → Connected` was always observable from outside. Reality: `RemoteAgentHostSessionsProvider`'s constructor sets `connectionStatus` to `Disconnected` (line ~134) and `tunnelAgentHost.contribution.ts._createProvider` calls `setConnectionStatus(Connecting)` *before* `registerProvider()` — both synchronously. By the time anything subscribes via `onDidChangeProviders`, transitions can already have raced past. First fix attempt used a `sawConnecting` flag and waited forever for `Connecting → Disconnected`; it never fired because the picker subscribed too late. **Prevented by:** new `gotcha` on [agent-host-sessions-providers](../../docs/agent-host-sessions-providers.md#debt--gotchas) documenting the synchronous status setup before registration and the racy observability from external subscribers.
- **`autorun` runs synchronously on creation.** The first `read(reader)` happens immediately, which means the autorun always observes the *current* status — not just future transitions. Without an `isFirstRun` guard, the autorun would synchronously trigger fallback on the initial `Disconnected` value before the connection attempt had a chance. **Prevented by:** captured in the change-summary key-decision; this is also baseline observable knowledge already covered by `instructions/observables.instructions.md`.
- **Tried to keep the policy in the view pane.** Moved logic back to the picker after several rounds of "still not working". The view pane's `onDidSelectWorkspace(undefined)` handler already did the right thing — duplicating the policy there made debugging harder and produced UI desync (picker showed restored selection while view pane created a copilot session). **Prevented by:** captured in the change-summary key-decision; the principle is "the component that owns the state should own the policy that mutates it".
- **Manual `setTimeout` + `toDisposable(clearTimeout)` instead of `disposableTimeout`.** Self-contained mistake, caught and fixed in the same conversation. **Prevented by:** general "prefer base/common/async helpers" rule already in `rob-coding-preferences.instructions.md` ("Helper Awareness").
- **Forgot to fire selection-change events when clearing on provider removal.** The `onDidChangeProviders` validate-on-removal path cleared `_selectedWorkspace` and updated the trigger label but didn't fire `onDidChangeSelection` / `onDidSelectWorkspace(undefined)` and didn't clear the connection-status watch — so `NewChatViewPane` could still think a workspace was selected and the grace-period timer kept ticking. Caught by Copilot code review. **Prevented by:** captured in the change-summary key-decision about action-signal vs render-signal events and in this summary.

## What we learned

- The local Agent Host provider already had an `authentication`-pending autorun in its constructor that drives eager session loading once auth settles. Remote providers don't have an equivalent and are visible to picker code in `Disconnected` state with no upcoming transition guarantee. The grace-period pattern fills that gap on the consumer side without requiring a provider-side contract change.
- `runWithFakedTimers({ useFakeTimers: true }, ...)` from `src/vs/base/test/common/timeTravelScheduler.js` plus `await timeout(N)` is the right pattern for testing time-driven fallbacks. Use it whenever a `disposableTimeout` is the unit under test.
- `_userHasPicked` is a small but load-bearing flag: it's the difference between "auto-restored selections can be silently upgraded" and "user-acknowledged selections are sticky". Anything that sets the selection without user intent (constructor restore, late provider upgrade) must NOT set it; anything driven by an explicit user gesture (`_selectProject`, `clearSelection`) must set it.
