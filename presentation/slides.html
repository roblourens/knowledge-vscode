Giving coding agents a memory
of the agent host

A small per-developer knowledge system, plugged into VS Code via skills

Rob · internal share-out

Part 1 — The problem

Where this came from

The agent host is a moving target

  • Big subsystem inside an even bigger codebase.
  • Touched by many contributors, including… my own coding agents.
  • Spans protocol, host process, remote transports, workbench chat integration, sessions app providers.
  • I am almost always working on it through an agent, in several concurrent sessions, sometimes with different models on the same task.

The thing that kept biting me

  • Every new session, the agent re-discovered the same context by reading code from scratch.
  • Hard-won decisions (“don’t do X here, we tried, here’s why”) died at the end of the conversation.
  • Architectural rationale lived in PR descriptions and my head — not anywhere the next agent would look.
  • Concurrent sessions had no way to learn from each other, and I didn’t want them to step on each other either.

Effectively: an onboarding problem, repeated dozens of times a week, with an amnesiac coworker.

What I tried first (and didn’t love)

Inline comments / READMEs in vscode

  • Pollutes the main repo with notes that are really my mental model.
  • PR review overhead for prose that isn’t product.
  • Drifts silently, no signal when it’s wrong.

Long system prompts / instructions files

  • Always-loaded ⇒ expensive and noisy.
  • No structure, no per-component scope.
  • No good story for “this changed last week”.

Part 2 — What I wanted in a solution

The shape of the answer, before any answer

Hard requirements

  • scope Lives outside the VS Code repo. Personal knowledge, not product.
  • access Reachable by agents with zero per-session setup from me.
  • concurrency Multiple sessions on multiple worktrees cannot clobber each other.
  • durability Survives across conversations, models, and machines (it’s a Git repo).
  • honesty Has an explicit story for staying in sync with the code — or admitting it isn’t.

Softer principles I cared about

  • Descriptive, not prescriptive. Docs describe how things are and why — not how they should be. More durable.
  • One concern per doc. A doc you can’t state the scope of in one sentence is too big.
  • Boring retrieval first. Index file + keyword search. No embeddings, no vector DB, until the boring thing actually fails.
  • Separate "what is" from "what happened" from "what we’re doing".
  • No friction. If I have to remember to run init, I’ll forget. Skills must self-bootstrap.

Things I deliberately chose not to optimize for

  • Multi-user / team-shared knowledge. One developer, one repo. Sharing can come later if it earns its way in.
  • Fancy retrieval. Index + grep is fine until it isn’t.
  • Auto-commit / auto-merge. I want to review the diff. Always.
  • A “framework”. This is plumbing for me, not a product.

Picking what to leave out was as important as picking what to build.

The drift problem deserved special attention

  • Any docs system about a fast-moving codebase becomes subtly wrong within weeks.
  • Wrong docs are worse than no docs — they confidently mislead the agent.
  • I wanted drift handling to be a first-class skill, not a vibe.
  • And I wanted it to be cheap: not “re-read every doc against all the code”.

Part 3 — The solution

A knowledge repo + a VS Code agent plugin that ships skills

Two things, one repo

  • A knowledge repo — docs/, tasks/, changes/, plan/, an index.md.
  • A VS Code agent plugin in the same repo, Team Kit–style, that registers a handful of skills under its own namespace.

The plugin’s skills are how agents (and I) drive the workflow. The repo is where the knowledge lives. They ship together because they evolve together.

Repo layout

vscode-knowledge/
├── index.md                         # entry point, hand-curated
├── docs/                            # how things ARE and WHY
│   ├── agent-host-topology.md
│   ├── agent-host-protocol.md
│   └── agent-host-session-handler.md
├── tasks/                           # how to DO recurring work
├── changes/                         # what HAPPENED, append-only
│   └── 2026-04-15-session-reconnect/
│       └── summary.md
└── plan/                            # ephemeral, per-session
		└── 2026-04-15-session-reconnect/
				├── plan.md
				└── tasks.md

Four directories — each answers a different question.

Anatomy of a doc

  • One component or concern. A Covers: line declaring the VS Code paths it’s about.
  • Describes the component, key files, patterns, and how it relates to neighbors.
  • Cross-links inline as plain relative Markdown.
  • Ends with a changelog: YYYY-MM-DD — <sha> — summary.
  • The most recent SHA is the baseline that reconcile diffs against.

Every doc starts life with an initial changelog entry. There is no doc without a baseline.

The skills

  • init — locate the repo, set up a per-session branch + worktree, symlink it into the VS Code worktree as .knowledge/, exclude it from Git.
  • plan — read relevant docs, write plan.md + tasks.md under plan/<date-slug>/.
  • implement — normal coding workflow, but with knowledge context loaded first.
  • finalize — distill the conversation into doc updates and a changes/ entry. Leaves the diff for me.
  • reconcile — detect drift since each doc’s baseline and fix the docs in place.
  • help — explain the workflow.

Auto-init: any skill that needs the repo bootstraps it silently.

Concurrency story

  • Each VS Code worktree gets its own knowledge worktree on its own branch.
  • Same VS Code branch run twice (e.g. two models, same task)? Each gets a disambiguated knowledge branch.
  • No two concurrent sessions ever share a knowledge checkout.
  • Merges happen at finalize time, on prose — rare and easy.

Concurrency safety falls out of Git, not out of clever locking.

How reconcile avoids being expensive

  1. Each doc has a baseline SHA (the last changelog entry).
  2. reconcile walks commits in origin/main since that SHA, intersected with the doc’s Covers: paths and code references.
  3. If nothing relevant changed ⇒ doc is presumed accurate, not re-read.
  4. If relevant changes exist ⇒ read them, update the doc in place, append a new changelog entry.

The output is an updated repo, not a drift report. Reports are a side effect.

Why this shape works for me

  • Lives where it belongs. Out of the VS Code repo, into a personal repo with its own lifecycle.
  • Loaded on demand. The agent pulls in only the docs the task touches.
  • Honest about staleness. Baselines + reconcile turn drift from a vibe into a workflow.
  • Plays well with how I already work. Worktrees, branches, PR review — all unchanged.
  • Cheap to start. Three docs and an index were enough to be useful.

What I’m watching for next

  • Does keyword + index retrieval hold up as docs/ grows past ~20 entries?
  • How often does reconcile actually catch real drift vs. noise?
  • Do changes/ entries earn their keep months later, or do they rot?
  • Is there a clean way to share a subset across the team without losing the “personal mental model” property?
