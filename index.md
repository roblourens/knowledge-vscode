# Agent Host Knowledge Index

The entry point for the VS Code agent host knowledge base. Read this file first to orient, then pull the specific docs relevant to your task.

## What this knowledge base covers

The VS Code **agent host** is a subsystem that hosts AI coding agents (Copilot, Claude, mocks, and remote/SSH/tunnel-relayed agents) inside VS Code. It owns:

- An **Agent Host Protocol (AHP)** — JSON-RPC + immutable state — that decouples the protocol layer from any specific agent or client.
- A **local Agent Host process** (utility process on desktop) that implements the server side of AHP for the built-in agents.
- **Remote Agent Host connections** over WebSocket, SSH relay, and tunnel relay.
- An **editor-window chat integration** under `/workbench/` that adapts AHP session state to VS Code chat sessions, file-edit checkpoints, terminals, language model selection, customizations, and auth.
- **Agent-window providers** under `/sessions/` that expose Agent Host sessions for both local and remote hosts.

Throughout this knowledge base, **agent window** means the `/sessions/` application surface and **editor window** means the `/workbench/` application surface. Source identifiers and historical changelog text may still use older product names.

Docs in this knowledge base are scoped to *one component or concern* each, with a `Covers:` line declaring the VS Code paths the doc is primarily concerned with. Use the `Covers:` lists below to find the right doc for a task; follow inline cross-references inside docs to reach adjacent context.

## Start here: design principles and topology

Before reading any component-level doc, internalize two things:

1. **The [design principles](docs/design-principles.md)** — the top-level agent-behavior guidance for protocol shape, Copilot/SDK evidence, pre-v1 breaking changes, state ownership, UI adapters, testing, and when to ask the user.
2. **The [topology](docs/agent-host-topology.md)** — where the code lives, how the layers relate, and the concrete decision tree for placing new code.

For the AHP-vs-ACP question ("why a new protocol?"), see the sibling repo's [docs/guide/ahp-and-acp.md](https://github.com/microsoft/agent-host-protocol/blob/main/docs/guide/ahp-and-acp.md): AHP is a coordination layer for N clients sharing a host; ACP is 1:1 communication between a client and an agent. They compose; today our first-party agents (Copilot, future Claude) sit in-process as `IAgent` implementations rather than via ACP.

## Major architectural layers

1. **Protocol layer** — `src/vs/platform/agentHost/common/state/`, plus the generated `state/protocol/` source synced from the sibling `agent-host-protocol` repo.
2. **Process and connection layer** — local utility process via MessagePort (`IAgentHostService`), remote WebSocket / SSH / tunnel connections (`IRemoteAgentHostService`), all behind `IAgentConnection`.
3. **Editor-window chat integration** — `AgentHostContribution`, `AgentHostSessionHandler`, `AgentHostEditingSession`, model and customization providers, under `src/vs/workbench/contrib/chat/browser/agentSessions/agentHost/`.
4. **Agent-window providers** — `LocalAgentHostSessionsProvider`, `RemoteAgentHostSessionsProvider`, and tunnel/SSH/WSL connection contributions under `src/vs/sessions/contrib/providers/`.

## Active debt & gotchas

Cross-cutting items only. Per-component items live in each doc's `## Debt & gotchas` section — always read it for any doc whose `Covers:` overlaps your task.

- **gotcha (system-notification live/replay symmetry)** — transcript-visible notifications have three coordinated paths: active-turn AHP response parts, idle `startsTurn` system-origin turns, and cold SDK-history reconstruction. Persisted replay must use root `assistant.turn_start` / `assistant.turn_end` boundaries because `session.idle` is not stored. See [agent-host-system-notifications](docs/agent-host-system-notifications.md#debt--gotchas).
- **gotcha (`_sessions` root-key invariant)** — `CopilotAgent._sessions` is keyed by **root** session IDs only and now stores `CopilotSessionEntry` values containing all chats for that session. Code resolving an entry from a subagent URI must walk the full `parseSubagentSessionUri` ancestry to root first. A single call is wrong for depth-2+ nested subagents. See [copilot-agent-provider](docs/copilot-agent-provider.md#debt--gotchas).
- **debt (`agentHostDiffs.ts` glue lacks direct tests)** — create/delete/rename/edit normalization is now centralized and unit-tested in `fileEditDiff.ts`, but the sessions-layer URI rebasing and `IChatSessionFileChange2` field wiring in `agentHostDiffs.ts` still have no dedicated test file. See [agent-host-git-driven-diffs](docs/agent-host-git-driven-diffs.md#debt--gotchas).
- **gotcha (layer rule for `vs/sessions/`)** — code under `src/vs/sessions/~` (i.e. `browser/`, `common/`, `node/` directly under `sessions/`) cannot import from `src/vs/workbench/contrib/*`; only `src/vs/sessions/contrib/<feature>/~` can. Enforced by `code-import-patterns` in `eslint.config.js`. New contrib folders under `src/vs/sessions/contrib/` must also be registered in `build/lib/i18n.resources.json`. See [agent-host-sessions-providers](docs/agent-host-sessions-providers.md#debt--gotchas) for the full entry.
- **debt (missing-working-directory repair is Copilot-only)** — the resume repair that recreates a missing worktree / resumes archived sessions read-only / surfaces the load error is implemented only in `CopilotAgent`. Claude and Codex still silently load an empty chat (no error) when a session's working directory is gone. The archived → read-only *enforcement* and *UI* are agent-agnostic, but the resume-side repair and error-throwing are not. See [copilot-agent-provider](docs/copilot-agent-provider.md#debt--gotchas).
- **gotcha/debt (agent-host archived state is server-owned; editor-window overlay must not mask it)** — for agent-host targets, archived truth is the `SessionStatus.IsArchived` bit, so `AgentSessionsModel.isArchived()` special-cases them to read `session.archived` and ignore the local view-state overlay (don't collapse it back into the generic `?? ` merge). Related debt: the editor-window viewer's Archive/Unarchive actions only write the local overlay and never dispatch `SessionIsArchivedChanged`, while the agent-window provider (`baseAgentHostSessionsProvider.archiveSession`) does. See [agent-host-session-handler](docs/agent-host-session-handler.md#debt--gotchas).
- **gotcha (AHP auth contract and turn preflight)** — agents with `protectedResources.required: true` MUST throw `AHP_AUTH_REQUIRED` (-32007) before authentication, not return empty results. Renderer turn dispatch preflights both normal and eager sessions; token forwarding dedupes/awaits in-flight RPCs with scoped generation invalidation, while missing Copilot credentials use focus-preserving product setup with structured cancellation/failure. See [agent-host-protocol](docs/agent-host-protocol.md#debt--gotchas), [agent-host-topology](docs/agent-host-topology.md#client-side-authentication-flow), and [agent-host-session-handler](docs/agent-host-session-handler.md#patterns-and-gotchas).
- **gotcha (Agent Host telemetry disablement)** — Agent Host telemetry uses a process-local, most-restrictive kill switch: startup/client `off` must permanently disable telemetry in that host process. Client levels propagate as root config string enum values, not numeric VS Code enums. See [agent-host-telemetry](docs/agent-host-telemetry.md#disablement-and-telemetry-level-propagation).
- **gotcha (no Agent Host `/untitled-*` resources)** — Agent Host-owned chat/session resources are final-looking from creation time; `/untitled-*` is only a generic contributed-chat staging URI and must be converted by `createNewChatSessionItem` before the Agent Host handler/provider owns it. The VS Code client chooses the AHP session URI and remote/local `createSession` must honor it. See [agent-host-session-handler](docs/agent-host-session-handler.md#chat-session-uri-ownership), [agent-host-sessions-providers](docs/agent-host-sessions-providers.md#new-session-uri-lifecycle), and [agent-host-remote-protocol-client](docs/agent-host-remote-protocol-client.md#session-creation-uri-ownership).
- **debt (remote transport semantics)** — soft reconnect/replay is implemented, but WebSocket, SSH relay, and tunnel relay close/error/dispose behavior still lacks a shared transport test matrix. See [agent-host-remote-protocol-client](docs/agent-host-remote-protocol-client.md#remaining-debt-candidates).
- **debt (Copilot CLI parity for selfhosting)** — Agent Host covers the core session substrate plus multi-chat, attachments, persistent MCP authentication, diffs, and feedback-tool display. Remaining parity pressure is narrower: optional turn-end commit/checkpoint lifecycle, MCP gateway/fallback/name-remapping behavior, plan-exit surfaces exposed only by the private SDK, and correlated request/OTel diagnostics. See [copilot-extension-host-cli](docs/copilot-extension-host-cli.md#parity-gaps-relevant-to-agent-host) and [copilot-sdk-tool-display](docs/copilot-sdk-tool-display.md#debt--gotchas).
- **debt (parallel session-metadata paths to changes view)** — agent-host populates the changes view's git fields via `_meta.git` → `setMeta` → `workspace.repositories[0]`, while Cloud / extension-host CLI populate the same fields via the workbench-side `IAgentSessionListController._buildMetadata` → `IAgentSession.metadata`. `changesViewModel.ts` currently fans both paths in with `?? workspaceRepository?.X` fallbacks. See [agent-host-sessions-providers#debt--gotchas](docs/agent-host-sessions-providers.md#debt--gotchas).
- **debt (AH chat decoration revival)** — AH chat requests are reconstructed from raw AHP state without re-parsing, so slash-command decorations (e.g. the skill pill) don't survive reload. Locally-persisted chat sessions don't have this problem because `reviveParsedChatRequest` restores the structural `IParsedChatRequest`. See [agent-host-customizations#decoration-revival-on-reload](docs/agent-host-customizations.md#decoration-revival-on-reload).
- **gotcha (workbench-setting registration is per-process)** — `chat.agentHost.*` keys registered only in `chat.shared.contribution.ts` are invisible to the main process and remote server (each process has its own `IConfigurationRegistry`). Settings consumed outside the renderer must be registered in platform-level `*.config.contribution.ts` files, side-effect-imported from the consumer; the server intentionally does not register `chat.agentHost.enabled` because it never reads it. The agent host process itself does not use workbench `IConfigurationService` — starters translate to env vars. See [agent-host-topology#debt--gotchas](docs/agent-host-topology.md#debt--gotchas).
- **gotcha (remote disconnect intent — tunnel vs SSH)** — both transports route user-initiated disconnect through `provider.disconnect()` → `_disconnectOnDemand`, but mean different things: tunnel persists suppression and keeps the cached tunnel; SSH removes the configured entry (no persisted suppression). For SSH, `removeRemoteAgentHost` MUST run before the SSH service's disconnect because `onDidChangeConnections` fires synchronously and triggers `_reconcile` → `_reconnectSSHEntries`. See [agent-host-sessions-providers#tunnel-backed-remote-providers](docs/agent-host-sessions-providers.md#tunnel-backed-remote-providers), [agent-host-sessions-providers#ssh-backed-remote-providers-disconnect-intent-is-removal](docs/agent-host-sessions-providers.md#ssh-backed-remote-providers-disconnect-intent-is-removal), and [remote-agent-host-management-ux#session-workspace-picker-inline-remove](docs/remote-agent-host-management-ux.md#session-workspace-picker-inline-remove).
- **gotcha (provisional worktree metadata ordering)** — a `Creating` Agent Host session still points at the selected checkout, not its final worktree. Git-state refreshes must be suppressed until materialization, `notify/sessionAdded` must upsert an existing provisional adapter with the final project/working directory, and list snapshots must overlay live workspace metadata. See [agent-host-git-driven-diffs#git-state-lifecycle-for-provisional-worktrees](docs/agent-host-git-driven-diffs.md#git-state-lifecycle-for-provisional-worktrees) and [agent-host-sessions-providers#new-session-uri-lifecycle](docs/agent-host-sessions-providers.md#new-session-uri-lifecycle).
- **debt (agent host spawns lazily / late at startup)** — the local agent host utility process only spawns on the first `IAgentHostService` access (`AgentHostContribution` at `WorkbenchPhase.AfterRestored`), one lifecycle phase after the extension host (`LifecyclePhase.Ready`); its persisted session cache also hydrates only when `LocalAgentHostContribution` constructs. Prewarming the process and reordering the contribution so cached/live sessions surface sooner is tracked in [#324330](https://github.com/microsoft/vscode/issues/324330). See [agent-host-topology#debt--gotchas](docs/agent-host-topology.md#debt--gotchas) and [agent-host-sessions-providers#startup-session-cache-cross-window-persistence](docs/agent-host-sessions-providers.md#startup-session-cache-cross-window-persistence).
- **debt (extension-host session refresh amplification)** — when its shared JSONL watcher is active, the extension-host CLI still combines targeted item updates with global list invalidation, while focus handlers can force all-provider refreshes without a visibility gate or freshness TTL. See [copilot-extension-host-cli](docs/copilot-extension-host-cli.md#debt--gotchas).
- **debt (stale extension-host worktree retries)** — provider refresh re-arms lazy resolution for observed rows; metadata that points at a deleted worktree can therefore retry Git repository opening indefinitely. See [copilot-extension-host-cli](docs/copilot-extension-host-cli.md#debt--gotchas).

## Recent changes

The `changes/` directory contains one subfolder per significant session, named `YYYY-MM-DD-short-description/`. Browse it directly for the narrative of how the system has evolved. The most recent few entries are usually the most useful context when working on adjacent areas.

## Conventions

- **Doc scope:** one component or concern per doc, with a `Covers:` line listing the VS Code paths the doc is primarily concerned with. If the `Covers:` list doesn't fit in a sentence, the doc is too broad — split it.
- **Cross-linking:** plain Markdown relative links inline where relevant. This index is the only file that tries to be exhaustive.
- **Changelog per doc:** every doc ends with a `Changelog` section. Entries are newest-first by date. Each entry: `**YYYY-MM-DD** — <short SHA> — <summary>`. Use a 10-character abbreviated SHA (e.g. via `git rev-parse --short=10`) — readable, and still unambiguous in practice. Insert new entries as the first bullet, above same-day entries and older entries; if backfilling an older entry, place it in date-sorted position. The first entry's SHA is the baseline that `reconcile` diffs against.
- **Debt & gotchas per doc:** every doc has a `## Debt & gotchas` section between the body and the changelog. Two kinds of entries, each one bullet line, dated, with the relevant file/symbol:
  - **`gotcha`** — "X is the way it is on purpose; if you touch it, do Y." Load-bearing weirdness; presumed permanent.
  - **`debt`** — "Y looks wrong / could be cleaned up / needs revisiting." Has an implicit lifetime; resolved when fixed. Example: `- **debt** (2026-04-17, agentSessionService.ts:registerProvider) — provider registration is duplicated in two code paths; should be unified once protocol v3 lands.`
- **Source of truth is the code.** When a doc and the code disagree, the code wins. Update the doc.
